1.2                Purpose & Goal

This specification addresses functional enhancements to the Configuration Back Office rules in IFS/Manufacturing.  These enhancements offer greater flexibility by incorporating a new ‘combination table’ object, providing a spreadsheet-like feature to derive values based on combinations of configuration factors.  This feature gives users a more compact and efficient means of maintaining configuration structures for product families with many variant components or operations.

Requirements come from Multiple Conditions-Actions Pairs, an idea specification by Baddis Boukehili for IFS client Helpman.

This functionality already exists in the form of code delivered against the 2001-3 code line, and thus should be treated as a type of a ‘merge’ project.

2                       Functionality

2.1                General

This process deals with the definition and execution, or evaluation, of back office configuration rules.  The existing conditions used to determine selection of an item is mostly unchanged by this design, except for a terminology change to ‘Selection Conditions’.

One enhancement to the CTO back office rules is covered by this specification:

·        Combination Table – A new back office rule object is created, conceptually similar to Configuration Formulas, allowing users to define a value result depending on combinations of other factors.  A combination table is referenced as an alternative way to derive values in conditions and actions.  ‘Combination Table’ is added to the list of value sources that currently includes ‘Characteristics Values’, ‘Characteristics Qty’, ‘Formulas’ and ‘Entered Values’.  A combination table is another form of IF/THEN logic, re-useable within a configuration family.

3.1.2             Activity Define Configuration Combination Table (new)

3.1.2.1       Functionality

A new navigator node, ‘Configuration Combination Table’ is added to the Configuration Back Office è Configuration Rules navigator folder.  Here users can define a re-useable object used to derive values based on various combinations of other factors.  A Combination Table is much like a Configuration Formula in concept, providing a means of defining a nested DECODE statement.

A combination table is defined as:

·        An ID and Description

·        Status – Either ‘Incomplete’ (indicating table is in process of editing) or ‘Valid’ (indicating factors cannot be changed).

·        Return Data Type (valid values either ‘Alpha’ or ‘Numeric’)

·        Configuration Family

·        Default Return Value can optionally be defined in the header.  If defined, this value is returned as an ‘ELSE’ if no combinations match the configuration.  Must match data type.

·        Note text can also be entered for each table.

For each table, users first define from 1 to 15 combination factors involved in the table: for example HEIGHT and LENGTH.  Factors can be of the following value types used in back office rules:  Characteristic Values, Characteristic Quantities, Formulas or other Combination Tables.  The same factor can appear more than once in the combination table, if desired (this allows a ‘BETWEEN’ capability, i.e. LENGTH > 10 and LENGTH < 20).  Users can only choose from among characteristics and formulas within the combination table’s family.  Users can optionally specify a ‘Test Value’ for each combination factor to be used when testing the table.

After factors are described, users can enter the data for the combination table.  At least one factor must be defined in order to create a combination value row.  For each row in the table, users define the following data:

·        Sequence – System generated, users can manipulate this to re-order the rows in the table.  This is used to break a tie if two combinations both evaluate to true.  If more than one row is true, the last combination in the sequence is returned.

·        Factor Operator – For each factor, users indicate the relational operator to use in this combination.  Valid values include: =, <, >, <=, >= and !=.  Default is ‘=’.  Required value.

·        Factor Value – For each factor, users enter the value for the factor in this combination.  Value must agree with the data type of the chosen factor type.  Required.  NULL is allowed here.  A NULL value indicates a match regardless of the value – in other words, a NULL value indicates this factor is not involved with this combination.  Users can specify the same factor more than once in a single combination table to allow range of value.

·        Resulting value – Each row in the table requires a resulting data value for this combination.  A required field, this value must match the data type of the combination table.  There is no other validation of this data; users are responsible for correct data in this column.

A ‘Test’ function is offered in the Combination Table client.  A test value must be provided for each factor in order for the ‘Test’ to be run.  Combinations are evaluated in sequence order.  In the event two combinations both evaluate to TRUE, the table returns a resulting value for the last combination in the sequence evaluating to true (JOMUUS: Can be debated.  This is how it worked for Helpman).  A NULL value in the value column acts as a wildcard (e.g. Height = 8 and Length = NULL is true for all Height = 8 values). NULL is returned if none of the combinations evaluate to true.

Deleting a factor in use with the table should cause the values in the associated CombinationValue attribute to be set to NULL for all of the table’s rows in the ConfigCombinationValue LU.

When choosing values of type Characteristic Value where the characteristic is a discrete option, the client should present a list of valid options to build the combination data.

Numeric data with decimals can be entered using any separator character (e.g. ‘,’ or ‘.’), however will be stored in US NLS format as a period (‘.’).  This is how decimals values are displayed.  This same issue is solved in existing Conditions and Actions value attributes.

3.1.3             Activity Define Configuration Formula

3.1.3.1       Functionality

Combination table data can be used as a new value type within Configuration Formulas. Thus ‘Combination’ joins the list of possible values in these clients.  Existing valid value sources are: Characteristic Value, Characteristic Quantity, Formula and Entered Value.

Example:             Characteristic Value BLOCK-HEIGHT = Combination BLOCK-PERM

If users choose to use a combination value, a list of values is presented including all combination tables existing for the relevant part’s configuration family.

3.2.3             Activity Evaluate Rules

3.2.3.1       Functionality

Configuration structures are evaluated from a variety of different applications including Interim Orders, DOP structures, and Shop Orders.  Whenever a configuration is evaluated, the application must adapt to return values from the new Combination Table object.  This section discusses the application server work associated with these changes.

Anywhere data values are used in Selection Conditions, Actions, Insert Action Attributes or Formulas, the application logic must be modified to also handle evaluation of combination tables.  In order to correctly evaluate the combination table, all combination factors must be present in the current configuration scope, with the exception of ‘Optional’ characteristics.  This logic is much the same as the evaluation logic associated with Configuration Formulas.

Combination tables are evaluated in a row-by-row manner ordered by combination sequence.  Each combination is parsed into a single logical statement evaluating to either TRUE or FALSE.  This is achieved by substituting all values for the current row with values obtained from the current configuration evaluation scope.

The Interim Demand Head window, tab ‘Result Log’ should be modified to enable a right-mouse button method to jump to the ‘Config Combination Table’ client for result log entries concerning combination tables.  The new column ‘CombinationId’ must be added to the result log LU to support this modification.  The column is populated for errors, warnings or information messages dealing with Combination Tables.
